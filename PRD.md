## Section 1 — App Overview

**Purpose**: Personal running and fitness dashboard for a single authenticated user. Syncs workouts from Apple Watch via iOS HealthKit, visualizes training load, manages race goals and training plans, and provides an AI coaching interface.

**Auth**: Google OAuth via Firebase Auth. A single owner email (`folstromjohn@gmail.com`) gates the HubBanner nav links.

**Hosting**: Vercel (production: https://training-web-rho.vercel.app)

## Section 3 — Data Model

All user collections live under `users/{uid}/…`. Exception: `stravaActivities` is a legacy global collection.

### `stravaActivities/{activityId}`
Legacy Strava data synced from iOS. **Not uid-scoped** (single-user legacy design).
Key fields: `id`, `name`, `type` (ActivityType), `start_date`, `distance_m`, `moving_time_s`, `avg_speed_mps`, `avg_heartrate`, `pace_sec_per_mile`, `gear_id`, `efficiencyScore`.

### `users/{uid}/healthWorkouts/{workoutId}`
Primary workout data written by iOS HealthKitSyncService.
Key fields: `workoutId`, `activityType` (raw HK string), `displayType` (human-readable), `startDate` (Timestamp), `endDate`, `durationSeconds`, `sourceName`, `isRunLike` (boolean, pre-computed on iOS), `hasRoute`, `routeComplete` (boolean | absent — iOS route-completion marker: `true` = full route written, `false` = partial/still completing, **absent on legacy docs ⇒ treated as complete**; web reads it as a display-hint only and never gates route availability on it), `hasHRStream` (boolean — true when iOS wrote a per-sample HR `hrStream` subcollection for this workout; absent/false ⇒ no stream), `calories`, `avgHeartRate`, `distanceMiles`, `distanceMeters`, `avgPaceSecPerMile`, `avgSpeedMPS`, `hrDriftPct`, `cadenceSPM`, `elevationGainM`, `prBadges` (string[]), `bestEfforts` (map keyed by `1mi`/`5k`/`10k`/`10mi`/`half` → `timeSeconds | null`), `trainingLoadV2` (number | null — Banister HR-reserve load, null renders "—"), `trainingLoadMethod` (`"streamed"` | `"avg-hr-fallback"` — which V2 path produced the score). `trainingLoadV2`/`trainingLoadMethod` are written client-side by `computeAndStoreTrainingLoad` via owner `setDoc(merge:true)` (same write path as `bestEfforts`; no new Firestore rule required). `weather` (map | absent) is a `WeatherSnapshot` — historical conditions at the run's start: `tempF`, `feelsLikeF`, `humidity`, `windMph`, `dewPointF`, `conditionText`, `conditionCode` (WMO code), `fetchedAt` (ISO). Fetched **client-side** from the Open-Meteo archive API by the run detail page when GPS exists and no snapshot is stored yet, then persisted via the same owner `setDoc(merge:true)` path (`saveWeatherForWorkout`; no new rule). Start coordinates come from the `route` subcollection's first point — there is **no `startLat`/`startLng`** on the doc. Fields `efficiencyRaw` and `efficiencyScore` are legacy; no UI reads them.

### `users/{uid}/healthWorkouts/{workoutId}/route/{pointIndex}`
GPS route points subcollection. Fields: `index`, `lat`, `lng`, `altitude`, `timestamp` (Timestamp), `speed`, `hr` (number | null — per-point heart rate in bpm, added by iOS sync, commit 84dfbf3; null when absent).

### `users/{uid}/healthWorkouts/{workoutId}/hrStream/{chunkIndex}`
Per-sample HR stream subcollection, written by iOS (MEA commit 3715519) for ALL workouts with in-window HR — gives non-route workouts (HIIT/OTF/strength) a true per-sample intensity basis. Chunk doc id is zero-padded `"%04d"`; doc shape `{ chunkIndex (Int), samples: [ { t (Timestamp), hr (Int bpm, never null in-stream) } ] }`. Chunks are globally time-ordered: read **ascending by `chunkIndex`** and concatenate each doc's `samples`. Parent doc's `hasHRStream === true` flags availability; a workout with zero in-window samples has no `hrStream` docs and `hasHRStream` is unset. Read web-side by `fetchHRStream` (`src/services/hrStream.ts`), mapping `{ t, hr }` → `{ timestamp: ISO, hr }`; returns `[]` when empty/absent (no throw). Consumed by (a) the streamed-load path (Section 5 item 4a) and (b) `WorkoutHRChart` — the full-workout HR line on `WorkoutDetailModal` (raw per-sample HR vs elapsed time; fetched only when `hasHRStream === true`, self-hides below 2 valid samples).

### `users/{uid}/healthWorkouts/{workoutId}/mileSplits/{mile}`
Per-mile split HR from iOS. Fields: `mile` (number, 1-indexed), `avgBpm`, `sampleCount` (read path requires `avgBpm && sampleCount >= 2` before using a split's HR). Security rule now in place (see Section 6 #10). The subcollection may still be empty for older runs — read paths in `runs/[id]/page.tsx`, `personal-insights/page.tsx`, and `RunActivityModal` (§6 #20) handle empty results gracefully.

### `users/{uid}/goals/{goalId}`
Custom running goals (`RunningGoal`, `src/types/goal.ts`; service `src/services/goals.ts`). Fields: `label` (string), `metric` (`'distance'` | `'time'` | `'count'`), `target` (number — miles / seconds / count, matching `metric`), `startDate` (ISO `'YYYY-MM-DD'` string), `endDate` (ISO string), `isActive` (boolean; soft-delete sets `false`), `createdAt` (Timestamp), `updatedAt` (Timestamp). Security rule: `allow read, write: if isOwner(uid)` — **must be added manually in the Firebase console** (see Section 6 #10). Progress is computed client-side via `computeGoalProgress` (Section 5).

### `users/{uid}/healthMetrics/{YYYY-MM-DD}`
Daily health snapshots (one doc per calendar date). Fields: `date`, `weight_lbs`, `bmi`, `resting_hr`, `steps`, `exercise_mins`, `move_calories`, `stand_hours`, `sleep_total_hours`, `sleep_awake_mins`, `sleep_start` (ISO string), `sleep_end` (ISO string), `brush_count`, `brush_avg_duration_mins`, `syncedAt`.

### `users/{uid}/healthMetrics/hourlyHeartRate`
Special singleton document (not a date-keyed record). Fields: `hourlyAvgBpm` (Record<"0"–"23", number>), `sampleCount`, `updatedAt`, `periodDays`.

### `users/{uid}/settings/prefs`
UserSettings singleton. Fields: `uid`, `displayName`, `email`, `weightThresholdGreen` (default 173 lbs), `weightThresholdYellow` (default 180 lbs), `defaultTargetPaceSecPerMile` (default 600 = 10:00/mi), `maxHeartRate` (optional user-set bpm), `restingHeartRate` (optional user-set bpm, 30–120, for the HR-reserve / Banister V2 load model; editable at `/settings`; resolved via `resolveRestingHr(settings)` with `DEFAULT_RESTING_HR=60` fallback), `thresholdPaceSecPerMile` (optional user-set threshold pace), `suggestedMaxHeartRate` (optional last computed bpm suggestion), `suggestedThresholdPaceSecPerMile` (optional prediction-based threshold suggestion), `suggestionsUpdatedAt` (Firestore Timestamp), `createdAt`, `updatedAt`.

### `users/{uid}/settings/healthGoals`
HealthGoals singleton. Optional per-metric goal objects: `weight` (WeightGoal with tolerance band), `bmi` (BMIGoal with min/max range), `restingHR`, `steps`, `sleep`, `brushing`, `exerciseMins`, `moveCalories`, `standHours`, `awakeMins`, `avgBrushMins` (all MetricGoal with `warningPct`/`dangerPct`). Field `updatedAt` is a Firestore serverTimestamp. **Not** the activity-ring goals — those live in the `healthGoals` collection below.

### `users/{uid}/healthGoals/{docId}`
Effective-dated activity-ring goal versions (`HealthGoalDoc`, `src/types/healthGoal.ts`; service `src/services/healthGoals.ts`). Fields: `effectiveFrom` (ISO `'YYYY-MM-DD'` — first date this version applies), `createdAt` (epoch ms; tie-breaker when versions share an `effectiveFrom`), `metrics` — one `DayOfWeekGoals` map (`{mon..sun}: number`) per ring metric: `steps`, `exercise_mins`, `move_calories`, `stand_hours`, `sleep_total_hours`. **Append-only history**: `saveHealthGoals` always writes a NEW doc (`crypto.randomUUID()` id, `stripUndefined`); versions are never overwritten or hard-deleted, so past days keep scoring against the goal active on those days. Resolution (`resolveGoalForDate`, `src/lib/ringMath.ts`): pick the latest `effectiveFrom <= date` (createdAt breaks ties), read that date's weekday value; with no qualifying doc, fall back to `DEFAULT_GOALS` (10,000 steps / 30 min / 500 kcal / 12 stand-hr / 8 sleep-h — provisional, pending product-owner confirmation). **Separate from** `users/{uid}/goals` (running goals) and `users/{uid}/settings/healthGoals` (threshold goals above). Security rule (`allow read, write: if isOwner(uid)`) is defined in the repo's canonical `firestore.rules` — it was **never actually published to the console** (a REST probe confirmed PERMISSION_DENIED), so goal saves fail until the repo rules file is deployed (see Section 6 #10).

### `users/{uid}/plans/{planId}`
Training plans. `planType` field discriminates: `"running"` (default when field absent) or `"workout"`. Entire plan in one document — no subcollections.
- Running: weeks[].entries[] are `PlannedRunEntry` (distanceMiles, paceTarget, runType, etc.). Optional `scheduledTime?` (24h `"HH:MM"` string) is display/export metadata only — set from the entry-form time input (blank input → `undefined` so `stripUndefined` drops it, never `""`); it adds time-of-day for calendar export and does **not** override the derived run date (still `startDate + weekIndex + weekday`). Workout entries do not carry it.
- Workout: weeks[].entries[] are `PlannedWorkoutEntry` (exercises[] with sets/reps/weight, or duration_mins for duration-only sessions). `category` field on entries enables category-aware auto-matching.
- Legacy `planType: "pilates"` docs are unsupported; code renders a delete prompt.
- **Lifecycle**: `status: "active" | "completed" | "draft"` is the in-app source of truth (`PlanStatus` in `src/types/plan.ts`). `draft` = not active and not completed. The legacy `isActive: boolean` is **kept on every doc as a mirror of `status === "active"`** because the iOS app reads `isActive` — every write sets both fields together and they never disagree. `completedAt?` (ISO string) is set only when `status === "completed"` — written by `setPlanCompletion(uid, plan, "complete")` and cleared on `"reopen"` (via `stripUndefined`, which drops the `undefined` field). Completion is **self-only**: it clears the plan's own active flag and never promotes/demotes a sibling. **Read-time derive (no write-back):** `fetchPlans`/`fetchPlan` compute `status` via `derivePlanStatus(data)` — explicit `status` wins, else legacy `isActive ? "active" : "draft"` (a legacy `isActive` never yields `"completed"`); they also normalize `isActive` from the resolved status so the in-memory object is consistent. Legacy docs are **not** rewritten on load — Firestore keeps the old shape until a normal edit re-saves the doc. `setActivePlan` dual-writes `{ status, isActive }` for same-type plans and **never demotes a `"completed"` sibling** (see `nextStatusForSibling`).

### `users/{uid}/halfMarathonRaces/{raceId}`
Race goals (general; name predates multi-distance support). Fields: `id`, `name`, `raceDate`, `raceDistance` (5K/10K/halfMarathon/marathon/custom), `customDistanceMiles`, `location`, `targetPaceSecondsPerMile`, `isActive`, `linkedPlanId`, `actualRunId`, `actualRunDate`, `actualRunDistanceMiles`, `actualRunDurationSeconds`, `actualRunAvgPace`.

### `users/{uid}/shoes/{shoeId}`
Running shoe inventory. ## Section 4 — Domain Invariants

1. **User scoping**: All data lives under `users/{uid}/…`. The only exception is `stravaActivities`, a legacy single-user global collection.
2. **Week boundaries**: Monday-start throughout. `weekStart()` in `src/utils/dates.ts` uses `(day + 6) % 7`. Matches iOS `WorkoutViewHelpers` convention.
3. **Workout overrides are non-destructive**: `workoutOverrides` docs layer on top of `healthWorkouts` at display time via `applyOverride()`. The source document is never written by the web app.
4. **planType backward compat**: Existing RunningPlan docs lack `planType`. `src/services/plans.ts` defaults to `"running"` on every read path. Never rely on `planType` being present.
5. **setActivePlan type isolation**: Activating a running plan deactivates only other running plans. Workout plans are independent. Uses atomic batch write.
6. **Plan auto-match (running)**: 4-pass best-fit matcher in `src/utils/planMatching.ts`. Tolerance = `max(0.5, plannedMiles × 0.3)`. A per-week `used` set prevents double-counting the same workout against multiple plan entries.
7. **Shoe assignment priority**: Manual assignments (in `shoeAssignments/manual`) always win over auto-assign rules. `evaluateAutoAssignRules()` skips any workout already in `existingAssignments` (including explicit `null`), so an explicit "no shoe" survives a matching auto-rule. Auto-assignments are derived (never persisted); both the listing and detail pages must resolve via `{ ...autoAssigned, ...manualMap }` — the detail page uses `useResolvedShoeAssignment` to stay in sync (a prior bug had it read only the manual map, so auto-assigned shoes never showed on the detail page).
8. **stripUndefined requirement**: All Firestore writes pass through `stripUndefined()` = `JSON.parse(JSON.stringify(obj))`. Firestore rejects `undefined` values.
9. **Auth guard**: All `(app)/*` routes require a valid Firebase Auth session. The `AuthGuard` component enforces this. No Firestore call should occur without a confirmed `uid`.
10. **Ring math** (`src/lib/ringMath.ts`, pure — no Firestore imports): Activity-ring progress is **uncapped** everywhere (1.42 = 142%; rendering wraps a slightly darker second arc past 100%). Ring goals are **effective-dated and per-day-of-week**: a day is always scored against the goal version active on that day (latest `effectiveFrom <= date`, `createdAt` breaks ties), so editing goals never re-scores history. Period progress is **sum-vs-sum**: Σ actual values ÷ Σ resolved per-day goals over the same range — never average-of-percentages or days-closed; days with no healthMetrics doc contribute 0 to the numerator while their goal still counts in the denominator. To-date periods (current week/month/YTD) never extend past today. "Today's" sleep = `sleep_total_hours` on today's doc (last night's sleep). Ring calculations never write to Firestore.

11. **Settings max HR is the single source of truth for client-side HR math** (product decision, 2026-06-11 audit): every surface that displays or uses a max heart rate — HR zone boundaries (Run Detail Zones card, Personal Insights "Running Intensity by HR Zone"), `TrainingLoadBadge` tooltips, and all live/in-memory training-load estimation — resolves `users/{uid}/settings/prefs.maxHeartRate` via `resolveMaxHr(settings)` (fallback `DEFAULT_MAX_HR = 185` only when unset) and passes the resolved value down. No display or calculation path may hardcode a max HR. Historical **persisted** `trainingLoadV2` values are never recomputed or migrated when the setting changes — only live computation follows the new value. (The iOS write-time pipeline is outside this repo's control; see MEA.)

12. **Date-only strings parse as LOCAL dates**: race (`halfMarathonRaces.raceDate`), plan (`plans.startDate`), and goal (`goals.startDate/endDate`) dates are stored as `"YYYY-MM-DD"` strings and MUST be parsed via `parseLocalDate` / counted via `daysUntil` (`src/utils/dates.ts`). `new Date("YYYY-MM-DD")` parses as UTC midnight, which renders the **previous** calendar day in US timezones and once made the same race show "Sat, Sep 5 · 86 days" on Plan Insights but "Sun, Sep 6 · 87 days" on Races. `daysUntil` counts whole local calendar days (0 = today), so the count never shifts with the time of day.

## Section 5 — Key Calculations & Business Logic

1. **Pace formatting** (`src/utils/pace.ts`): `formatPace(secPerMile)` rounds total seconds first (`Math.round(secPerMile)`), then extracts minutes/seconds with integer arithmetic. Fixes the "10:60" rounding bug. Same pattern in `formatDuration`.

2. **HR Drift** (`src/utils/metrics.ts`): `((secondHalfAvgHR - firstHalfAvgHR) / firstHalfAvgHR) × 100`. Computed on device; requires ≥20 min or ≥2.0 mi. Thresholds by distance bucket — short (<3 mi): good ≤5%, ok ≤10%; medium (3–6 mi): good ≤7%, ok ≤12%; long (6+ mi): good ≤10%, ok ≤15%.

3. **Cadence** (`src/utils/metrics.ts`): Thresholds — short: good ≥170 spm, ok ≥160; medium: good ≥168, ok ≥158; long: good ≥165, ok ≥155.

4. **Training Load (TRIMP) — LEGACY zone-multiplier model, RETIRED FROM DISPLAY (Prompt 3)** (`src/utils/trainingLoad.ts`): `computeTrainingLoad()` is marked `@deprecated` and is no longer referenced by any display/aggregation path (kept in place, unreferenced, for trivial rollback; remove in a future cleanup). All load surfaces now go through `resolveDisplayLoad()` (see item 4a). Description of the legacy model retained for reference:
   - Formula: `TRIMP = durationMinutes × zoneMultiplier(avgHR) × factor`
   - `DEFAULT_MAX_HR = 185 bpm` fallback. Consumers resolve `users/{uid}/settings/prefs.maxHeartRate` via `resolveMaxHr(settings)` and pass the resolved value into `computeTrainingLoad()`, `buildDailyLoadMap()`, zone-boundary helpers, badges, dashboard/insights summaries, and AI Coach context.
   - Zone multipliers (running zones): Z1 Recovery <60% ×1.0 | Z2 Aerobic 60–70% ×1.5 | Z3 Tempo 70–80% ×2.5 | Z4 Threshold 80–90% ×4.0 | Z5 Max ≥90% ×6.5
   - Strength activities use shifted zone bands (Z5 starts at 80%, not 90%)
   - Post-TRIMP scaling factors: running/cardio = 1.0 | **HIIT/OTF = the session's average HR reserve** (intensity-proportional, `clamp(avg HRR, 0, 1)`; `DEFAULT_HIIT_FACTOR = 0.75` fallback only when avgHR/reserve is unusable) | strength (lifting/cooldown) = 0.25 | mindful (yoga/pilates/barre) = 0.20. Extracted into the shared `activityLoadFactor(activityType, hrr?)` helper (the optional `hrr` is consumed ONLY by the HIIT/OTF branch; running/strength/mindful ignore it) so the legacy and V2 models stay in sync. **HIIT was a flat 0.75**, but validation against Strava RE showed a flat factor cannot fit (easy HIIT wants ~0.62, hard ~0.89, tracking avg %HRR ~1:1); `f = avg HRR` is parameter-free and collapsed mean error from ~15% to ~4% across 6 reference sessions (**small ground-truth sample — relationship + 0.75 fallback are TUNABLE**). HIIT/OTF ONLY: running/strength/mindful factors and the shared `bannisterWeight`/HRR curve are unchanged (runs validate at factor 1.0 on that curve, so steepening it would regress them). For the streamed path the factor uses one per-session scalar (avg HRR from `avgHeartRate`), NOT a per-sample factor — the per-sample integral remains the base load.
   - `STRENGTH_LOAD_FACTOR` was bumped from 0.20 → 0.25 (validated against Strava Relative Effort)
   - `computeTrainingLoad()` returns null when HR or duration is missing/invalid

4a. **Training Load V2 — Banister TRIMP on HR reserve** (`src/utils/trainingLoad.ts`, `computeTrainingLoadV2`): the universal avg-HR baseline that will score ALL runs after the Prompt-3 cutover (a streamed per-second override arrives in a later phase). Additive — the legacy item 4 model is untouched. Math: `HRR = clamp((avgHR − restingHr) / (maxHr − restingHr), 0, 1)`; `weight = 0.64·e^(1.92·HRR)` (Banister male coefficients, a tunable pair); `rawTrimp = durationMinutes · HRR · weight`; `load = round(rawTrimp · TRAINING_LOAD_V2_SCALE · activityLoadFactor)`. Anchors/tunables: `maxHr` via `resolveMaxHr` (fallback `DEFAULT_MAX_HR=185`), `restingHr` via `resolveRestingHr` (fallback `DEFAULT_RESTING_HR=60`), `TRAINING_LOAD_V2_SCALE=1.14` (anchored so validated reference runs ≈ Strava Relative Effort). Returns null (never 0) when avgHR is missing/invalid, duration ≤ 0, or `maxHr ≤ restingHr` (divide-by-zero guard). Validated reference runs (maxHr 164, restingHr 60): easy 137 bpm/1854 s → 69, hard 156 bpm/1764 s → ~116, long 152 bpm/4370 s → ~256.
   - **Streamed integral** (`computeStreamedTrainingLoad`): a precision OVERRIDE for dense-HR runs **and HIIT-style hrStream workouts** that integrates the Banister weighting over every consecutive per-point timestamp pair instead of using one whole-run HRR. `hrCoverage` = points-with-finite-hr / total; if `hrCoverage < STREAMED_HR_COVERAGE_MIN=0.5` or `< 2` points it falls back to the avg-HR model (`method: "avg-hr-fallback"`). Per step: `dt = t[i+1]−t[i]`; `dt ≤ 0` skipped; `dt > TRAINING_LOAD_DT_CLAMP_SEC=10` is **clamped** (pause/auto-pause gaps still count, capped — not dropped); `hr` carries forward the last valid sample; `rawStep = (dt/60)·hrr·bannisterWeight(hrr)`. Reuses the SAME `bannisterWeight` / `TRAINING_LOAD_V2_SCALE` / `activityLoadFactor` as the avg-HR model (no duplicated constants). A clean ~1 Hz stream reconciles with the avg-HR result within ~5% for steady efforts, but **spiky HIIT reads materially higher** — `hrr·bannisterWeight(hrr)` is convex, so the per-sample integral credits time at HR spikes that a whole-run average flattens away (this convexity gain is the motivation for consuming `hrStream`). Same null guards (`maxHr ≤ restingHr` → null). `TRAINING_LOAD_DT_CLAMP_SEC` and `STREAMED_HR_COVERAGE_MIN` are tunables.
   - **Store + backfill** (`src/services/healthWorkouts.ts`): `computeAndStoreTrainingLoad(uid, workoutId, settings)` resolves `resolveMaxHr`/`resolveRestingHr`, then selects the load model via a **3-tier method-selection chain → route → hrStream → avg-HR**: (1) `hasRoute` → `computeStreamedTrainingLoad` over route-point HR (UNCHANGED run path, `method: "streamed"`); (2) else `hasHRStream` → `fetchHRStream(uid, workoutId)` and, when it returns `≥ MIN_HRSTREAM_SAMPLES=2` samples, `computeStreamedTrainingLoad` over those per-sample points (`method: "streamed"` — same integral, the hrStream samples ARE the points array); (3) else `computeTrainingLoadV2` from avg HR (`method: "avg-hr-fallback"`). An empty/too-sparse `hrStream` despite `hasHRStream` falls through to avg-HR (defensive — never errors). Both `hrStream`-scored and route-scored loads surface as `method: "streamed"`, so the tooltip reads "Calculated from your second-by-second heart rate" for both. Merges `{ trainingLoadV2, trainingLoadMethod }` back (never writes 0 for a null load — null is preserved so the UI shows "—"). `backfillTrainingLoad(uid, settings)` iterates all run-like workouts, computes+stores each, and tallies `{ processed, streamed, fallback, skipped }`; idempotent (overwrites).
   - **Settings-triggered recompute — source-of-truth realignment for stored load** (`recomputeAllTrainingLoad(uid, settings)` in `src/services/healthWorkouts.ts`; change-detection `hrAnchorsChanged(prev, next)` in `src/utils/trainingLoad.ts`): stored `trainingLoadV2` is written against the HR anchors in effect at compute time, so changing `maxHeartRate`/`restingHeartRate` later leaves every stored value stale (computed on the old reserve) until realigned. The Settings page closes this automatically: after `saveUserSettings` succeeds, `handleSave` diffs the just-saved `{maxHeartRate, restingHeartRate}` against a `prevAnchorsRef` snapshot (seeded on load, advanced **after** each save attempt) via `hrAnchorsChanged` (true if EITHER anchor changed, incl. `undefined`↔number). On change it runs `recomputeAllTrainingLoad` over **ALL HR-bearing workouts of ANY type** (`hasRoute || hasHRStream || finite positive avgHeartRate`; **deliberately NO `isRunLike` filter** — unlike the runs-only `backfillTrainingLoad` — so HIIT/OTF/strength/Pilates, where the avg-HRR factor matters most, are included), calling the **verbatim** `computeAndStoreTrainingLoad` once per workout (NO new math; same 3-tier chain; same `merge:true` write of `{trainingLoadV2, trainingLoadMethod}` only; no-HR docs skipped). UI: a "Recomputing training loads…" state (`text-textSecondary`), Save disabled in-flight, then "Updated N workouts" (`text-success`); recompute failure is **non-blocking** (settings stay saved; idempotent — safe to re-save to retry) and `prevAnchorsRef` advances only after the attempt so a failure re-triggers next save. Because all surfaces read stored load through `resolveDisplayLoad`, this realigns This Week, Runs, Workouts, Personal Insights, and Plan Insights with no per-page change.
   - **Backfill script** (`scripts/backfillTrainingLoad.ts`, driven by the env-gated Vitest entry `scripts/backfillTrainingLoad.test.ts` since the repo has no tsx/ts-node): computes & stores `trainingLoadV2`/`trainingLoadMethod` for **all HR-bearing workouts in the last 12 months** (not runs-only — widened so the hrStream/streamed scores reach HIIT/OTF/strength) using the same **3-tier chain (route → hrStream → avg-HR)** as `computeAndStoreTrainingLoad`, reusing the exact `computeStreamedTrainingLoad`/`computeTrainingLoadV2` exports via the admin SDK. **Selection**: the Firestore query is the **date window only** (`startDate >= now − 365d`, newest-first); the `hasRoute || hasHRStream || finite positive avgHeartRate` HR-basis filter is applied **in memory** (Firestore can't OR those field predicates with the date inequality). Workouts with no basis (could only yield a null load) are excluded and counted as `skipped(no-HR-basis)`. **DRY-RUN by default** (`BACKFILL=1 npx vitest run scripts/backfillTrainingLoad.test.ts` — prints a per-workout table + a 4-way method breakdown `streamed(route)` / `streamed(hrStream)` / `avg-hr-fallback` / `skipped(no-HR-basis)`, writes nothing); **`BACKFILL=commit`** performs batched owner `setDoc(merge:true)` writes to all matched types. Null loads are SKIPPED (never written as 0 → UI falls back to "—"). Resting HR falls back to `DEFAULT_RESTING_HR=60` (logged). Idempotent. The default `npm test` skips the runner (no Firestore).
   - **Enrich-on-load — automatic store/upgrade when workouts load** (`enrichTrainingLoads(uid, workouts, settings)` in `src/services/healthWorkouts.ts`; pure predicate `shouldEnrichLoad` + loop key `enrichBasisKey` in `src/utils/trainingLoad.ts`; trigger hook `src/hooks/useEnrichTrainingLoads.ts`): closes the gap that iOS writes each `healthWorkouts` doc with NO load — and in two phases (base doc first, then `avgHeartRate`/`hrStream` in a LATER write) — while `computeAndStoreTrainingLoad` had no ingest/on-view caller, so a newly-synced workout's STORED `trainingLoadV2` stayed null until a manual trigger (the settings-recompute or the backfill script). Now, after a page's workout list resolves, `enrichTrainingLoads` filters the loaded set by `shouldEnrichLoad` and calls the **verbatim** `computeAndStoreTrainingLoad` once per match (sequential/concurrency-1, idempotent, returns the count written; NO new math, same 3-tier chain, same `merge:true` write of `{trainingLoadV2, trainingLoadMethod}` only → **no Firestore-rules change**). `shouldEnrichLoad` fires for two reasons: **(a) STORE** — no finite stored load yet AND the workout has an HR basis (`hasRoute || hasHRStream || finite positive avgHeartRate`); **(b) UPGRADE** — stored via `"avg-hr-fallback"` AND a richer basis (`hasRoute || hasHRStream`) has SINCE arrived (re-run upgrades the method to `"streamed"`). It returns false — correctly leaving the live "—"/avg-HR display untouched — when there's no HR basis yet, or the value is already `"streamed"`. **Trigger surfaces**: `useEnrichTrainingLoads(uid, workouts, settings)` runs in an after-paint effect (never blocks render) on the **Workouts page over its non-run set** (`allWorkouts`; that page subscribes `isRunLike:false`) AND the **Runs page over its runs** (`allRuns`) — together these cover the full collection, since no single page loads both. Writes flow back through each page's own `onSnapshot` listener and re-render via `resolveDisplayLoad`. See §6 #22 for the store-early-then-upgrade behavior and the snapshot-loop / settling guards.
   - **Display resolver + cutover (Prompt 3)**: `resolveDisplayLoad(workout, maxHr, restingHr)` is the SINGLE entry point every load surface uses. Stored `trainingLoadV2` (finite number) wins; otherwise it live-computes the avg-HR `computeTrainingLoadV2` baseline (NEVER legacy, NEVER streamed — streamed only ever comes from the stored field); null when there's no stored value and no usable avg HR (caller renders "—"). All display/aggregation paths were cut over: runs/workouts/dashboard/plan-insights/personal-insights pages, `buildDailyLoadMap` (acute/chronic/ACWR), `TrainingLoadBadge` (now takes a precomputed `score` prop), AI Coach context. Weekly sums filter nulls before reducing (never coerce to 0). **Until `backfillTrainingLoad` is run, no run has a stored `trainingLoadV2`, so every run shows its LIVE avg-HR V2 value; dense-HR runs will only read their (higher) streamed value after the backfill.** Expect the app-wide load scale to drop materially vs the legacy model (e.g. ~602 vs ~1148 weekly). Settings: `restingHeartRate` input added at `/settings` (30–120 bpm, default 60).
   - **Tooltip explainer**: `TrainingLoadBadge`'s hover tooltip appends a per-run "How this was calculated" section built by the pure `buildLoadExplainer(inputs)` helper — shows the method (streamed = "second-by-second heart rate" / fallback = "average heart rate", with a "(live estimate)" tag when no stored method), the HR reserve used (`round(HRR×100)%` with the resting–max anchors), avg HR, duration, the load score, and a one-line Banister-TRIMP note. Inputs are optional props (`durationSeconds`, `restingHr`, `trainingLoadMethod`); when absent or insufficient the section is omitted (no NaN, no blank "%").

5. **Training load ratio** (`src/utils/metrics.ts`): `ratio = acute7d / chronic30d`. Thresholds: <0.8 = "deload" | 0.8–1.1 = "stable" | 1.1–1.4 = "building" | >1.4 = "aggressive".

6. **CTL/ATL/TSB (PMC)** (`src/utils/trainingLoadSeries.ts`): EWMA with `CTL_DAYS=42` (fitness), `ATL_DAYS=7` (fatigue), `TSB=CTL−ATL` (form). α = 1 − exp(−1/τ). Rest days fill with load=0. Seed CTL/ATL at 0; pass a start date ≥3×CTL_DAYS before display window for convergence.

7. **Riegel race prediction** (`src/utils/riegelFit.ts`): Weighted WLS regression in log-log space: `ln(time) = a + k × ln(distance)`. Recency: 5-week half-life (0.5^(days/35)). Tier weights: RACE=3.0, QUALITY=1.75, BASELINE=1.0. Individual weights normalized then capped at 5.0. Half marathon gate: ≥4 efforts + 2+ medium-long runs (≥4 mi in last 35 days) + longest ≥6 mi — bypassed when a RACE-tier anchor ≥ target distance exists. Sanity filters: pace 4:30–15:00/mi, ≥0.5 mi, ≥5 min. **`asOf` parameter** (`buildQualifyingEfforts({ asOf })`, default = wall-clock now): the reference date for BOTH the lookback windows (56d ordinary / 120d race) AND each effort's `ageDays` (which drives the recency decay). Runs after `asOf` are excluded. Passing a past `asOf` recomputes a prediction "as of" that date with the SAME model — weights, k-clamp, decay formula are unchanged; only the reference date moves (`asOf=now` reproduces the prior value exactly). The Plan Insights card and the weekly trend both go through the shared pure `predictRaceTime(runs, params, asOf)` (`src/utils/racePrediction.ts`) so they can never diverge.

7a. **Weekly prediction trend** (`buildPredictionTrend`, `src/utils/racePrediction.ts`): for each plan week, calls `predictRaceTime` with `asOf` = that week's END (Mon-start + 6d, end-of-day — the SAME boundaries `buildPlanAdherence` uses, so the trend lines up week-for-week with the adherence charts). Future weeks (start > now) → `predictedSeconds: null`; the in-progress week uses `asOf = now` so its point equals the live card; weeks with insufficient data → `null` (the line breaks rather than fabricating a point). `goalSeconds` (target pace × distance, or null) is a constant on every point for the reference line. Pure / in-memory — no Firestore, no new storage.

8. **Mile split segmentation** (`src/utils/mileSplits.ts`): Cumulative haversine distance over GPS route points. Timestamps interpolated linearly at mile boundaries. Final partial mile skipped if <0.05 mi.

9. **PR computation** (`src/utils/prComputation.ts`): Pace = `durationSeconds / distanceMiles` (not the stored `avgPaceSecPerMile`, which can be null on older sync rows). Band PRs and specific-distance PRs (5K, 5 mi, 10K, 15K, 10 mi, HM).

10. **Shoe auto-assign** (`src/utils/shoeAutoAssign.ts`): Rule specificity score: scope≠"any" +10, minDistance +5, maxDistance +5, startDate +3, endDate +3. Highest score wins. Manual assignments always take precedence.

11. **Plan Insights "Recent Trends" window** (`src/app/(app)/plan-insights/page.tsx`): All three Recent Trends KPIs (total miles + run count, avg run distance, longest run) share one cutoff/label derived from the selected race's linked plan. Resolution order: (1) `activePlan.startDate`; (2) earliest week's `startDateLabel` parsed as `"<label> <currentYear>"` (defensive — not on the `PlanWeek` type, read via cast); (3) 30-day fallback with a `console.warn`. Label is `Since plan start · <Mon D>` when resolved, else `Last 30 days`. Previously each KPI used an independent hardcoded 30-day (longest run: 56-day) lookback. The KPIs filter a slice of the already-loaded `runs` array — no extra Firestore read.

12. **Grade Adjusted Pace (GAP)** (`src/utils/gradeAdjustedPace.ts`): **GAP is now RATIO-BASED** — the run-level KPI = `avgPaceSecPerMile × (1 / aggregateFactor)`, i.e. the trusted device pace scaled by a unitless grade ratio. This fixes a systematic GAP-slower-than-pace bug on net-gain runs caused by three issues: (#1) **basis mismatch** — the displayed Pace KPI is the device `avgPaceSecPerMile`, but GAP was previously recomputed on GPS 2D-haversine distance (which runs shorter than device distance → GAP slower even on flat ground); (#2) **Jensen convexity** — summing per-segment `1/factor` over symmetric rolling terrain inflates GAP slow (1/factor is convex); (#3) **dead-band** — the ±1.5% dead-band zeroed gentle sustained net climbs (avg grade often sub-1.5%). The `aggregateFactor` is ONE Minetti factor (`gradeAdjustmentFactor`, Minetti et al. 2002, grade clamped ±30%, floored 0.1) computed from the run's **net smoothed elevation change over total moving horizontal distance** — so symmetric rolling (net ≈ 0) yields ratio ≈ 1 (no slow-bias) and flat ground gives GAP == displayed Pace exactly, while net-uphill → faster and net-downhill → slower. Inputs preserved: 11-point centered altitude smoothing (`ALT_SMOOTHING_WINDOW=11`), the 25 m resampled baseline (`GRADE_BASELINE_METERS=25`), and the ±1.5% dead-band (`GRADE_DEADBAND_PERCENT=1.5`) — but the dead-band now applies **only to per-span noise rejection feeding the chart/table series**, NOT to the run-level aggregate grade. **Moving time/distance** are derived from route points (a segment is moving only if it covers ≥`MIN_MOVING_DIST_M=1.0` m at ≥`MIN_MOVING_SPEED_MS=0.5` m/s); the GPS-derived per-point and per-mile GAP series are scaled by a single basis-correction (`avgPaceSecPerMile / gpsRawMovingPace`) so they read consistently with the KPI. When `avgPaceSecPerMile` is null, the basis falls back to GPS raw moving pace. `computeRunGap(points, totalDistanceMiles, movingTimeSec, avgPaceSecPerMile?)` also returns `netRiseM` (NET elevation change in meters, end−start, endpoint-averaged; null when no geometry) for the Run Detail elevation KPI. **Net-grade hardening:** (a) the net rise is averaged over the first/last `NET_GRADE_ENDPOINT_SAMPLES=5` moving points' smoothed altitudes (single-endpoint fallback when fewer than 2× that many moving points), so one noisy first/last GPS sample can't tilt the whole-run net; (b) a tight aggregate-grade dead-band `AGGREGATE_GRADE_DEADBAND_PERCENT=0.10` snaps pure endpoint-noise jitter (|net grade| ≤ 0.10%) to flat (GAP == pace) while leaving real shallow descents intact (e.g. the −0.229% reference run stays ~+7 sec slower). **0.10% is a tunable** (measured real-vs-noise endpoint variance ≈ 0.02%, well below it).

13. **Goal progress** (`src/utils/goalProgress.ts`): `computeGoalProgress(goal, runs, today) → GoalProgress`. Status by date range: `upcoming` (today < startDate) / `active` / `completed` (today > endDate). `actual` sums each in-range run's `distanceMiles` (distance), `durationSeconds` (time), or +1 (count). Pace status (active only): `ahead` when actual ≥ 102% of expected linear pace (`target × daysElapsed/daysTotal`), `behind` when ≤ 98%, else `on_track`; completed goals report met/missed via `percent`. Time targets are stored in seconds (the form converts hours→seconds on save).

14. **Rolling average** (`src/utils/smoothSeries.ts`): `rollingAverage(values, windowSeconds, timestampsSec)` — time-windowed centered mean (default window `SMOOTH_WINDOW_SEC=25`). Nulls are preserved (window with no finite values → null) so line breaks survive; falls back to a fixed-count window when timestamps are unusable. Used to smooth the pace and GAP display series in `RunOverlayChart` (applied after outlier-nulling; underlying GAP/per-mile values untouched).

15. **Pace axis domain** (`src/utils/paceAxisDomain.ts`): `computePaceAxisDomain(values)` builds a robust y-axis [min,max] from the p5/p95 percentiles of finite pace/GAP values with padding, clamped to `MIN_PACE_FLOOR_SEC=240` (4:00/mi) – `MAX_PACE_CEIL_SEC=1200` (20:00/mi) so GPS-glitch spikes don't crush the real signal band. `nullifyOutliers(values, domain)` returns a display copy with out-of-domain values set to null (Recharts draws a line break instead of a clamped spike). Display-only — neither alters any pace/GAP computation.

16. **Plan adherence aggregation** (`src/utils/planAdherence.ts`): `buildPlanAdherence(plan, runs, { maxHr, restingHr?, throughDate? })` is the **single source** for both Plan Insights (passes `throughDate = getWeekStart(now)` → "elapsed weeks only", reproducing the prior inline behavior exactly) and the completion summary (omits `throughDate` → **full span**). Per week it returns planned/actual miles, planned/completed runs, `runLoad` (Σ `resolveDisplayLoad`, null-HR runs excluded), and weighted avg pace (Σ duration ÷ Σ miles; null when no runs); plus span totals and `weeksHitTarget` (week where actual ≥ 85% planned). Reuses `matchPlanToActual` for ±1-day matching + bonus-run inclusion — no reimplemented match/load/pace math. Pure (no Firestore/React). **Workout plan summary** (`src/utils/workoutPlanSummary.ts`): `buildWorkoutPlanSummary(plan)` counts planned (`type !== "rest"`) vs completed (`entry.completed === true`) overall and by category — OT (`category === "orangetheory"`), Pilates (`"pilates"`), and an **Uncategorized** bucket (non-rest, no `category`) surfaced rather than hidden — plus per-week session counts. Completion is read from the entry, so no activity pool is needed. Both feed `PlanCompletionSummary` (collapsible, rendered at the top of the plan detail only when `status === "completed"`).

16. **Pace-by-distance trend** (`src/lib/paceRangeTrend.ts`): `computePaceRangeTrend(runs, minMiles, maxMiles, window, now)` powers the Personal Insights "Pace by distance" section. A run qualifies when its **total** `distanceMiles` is within `[minMiles, maxMiles]` inclusive AND `date >= windowStartDate(window, now)`. Per-period avg pace is **distance-weighted**: `sum(durationSeconds)/sum(distanceMiles)` within the bucket (never the arithmetic mean of per-run paces). Granularity: `1m/2m/3m → week` (Monday-start via `weekStart`), `6m/12m/ytd → month` (month-start computed inline). `windowStartDate`: `ytd → Jan 1 of now's year`, `Nm → now minus N calendar months`. Validity guard drops runs with `distanceMiles ≤ 0`, non-finite pace, or pace outside `[180, 1200]` sec/mi. Returns raw seconds; the section formats with `formatPaceLabel`.

17. **Best Efforts** (`src/utils/bestEfforts.ts`): `computeBestEfforts(points)` computes each run's fastest continuous elapsed-time segment for `1mi`, `5k`, `10k`, `10mi`, and `half`. It builds cumulative 2D haversine distance and elapsed-time arrays from route points, then uses an O(n) two-pointer sliding window per target distance. The right boundary is linearly interpolated when the window overshoots the target distance (`t = t0 + ((D - d0) / (d1 - d0)) * (t1 - t0)`) so results represent exactly the standard distance, not the next sampled route point. Stopped/micro-pause time is intentionally included, matching Strava-style elapsed best efforts; GAP's moving-time stop filtering is not used here. Runs shorter than a target distance store `null` for that key. Results are persisted on `users/{uid}/healthWorkouts/{workoutId}.bestEfforts`; the idempotent backfill service skips existing `bestEfforts` docs before route points are fetched.

18. **Athlete Profile suggestions** (`src/utils/maxHrSuggestion.ts`, `src/utils/thresholdPaceSuggestion.ts`): Max-HR suggestion is an explicit user-triggered calculation, not a page-load hot path. The service reads up to 30 recent GPS run route subcollections, collects per-point `hr`, filters values outside 40–220 bpm, requires at least 50 valid samples, and returns the rounded 99th percentile. Threshold-pace suggestion reuses the existing Riegel model output: predicted 10-mile pace is preferred (`predictSeconds(fitTen, 10.0) / 10`), with predicted half-marathon pace as fallback. Treating 10-mile predicted pace as threshold pace is a domain assumption to validate against user preference.

19. **Pace Zones** (`src/utils/zones.ts`, `src/components/ZoneBreakdown.tsx`): Run Detail shows threshold-based pace zones below HR zones when `users/{uid}/settings/prefs.thresholdPaceSecPerMile` is set. The model uses actual per-point pace (not GAP) and attributes elapsed time between consecutive route points to the earlier point's pace. Zone bands use pace/threshold ratio, where ratio > 1 is slower than threshold and exact threshold pace is Z3: Z1 Recovery ≥1.20, Z2 Easy 1.10–1.20, Z3 Threshold 0.97–1.10, Z4 Interval 0.90–0.97, Z5 Repetition <0.90. Invalid/null pace and spikes >1800 sec/mi are excluded. When threshold pace is unset, Run Detail shows a settings prompt instead of a run-relative fake bar; when set, the Pace Zones info tooltip shows the actual pace ranges derived from the user's threshold.

20. **Shoe replacement projection** (`src/utils/shoeProjection.ts`): `projectShoeReplacement({ currentMiles, limit }, shoeRuns, now?)` estimates a shoe's remaining life from its RECENT rate. `recentMilesPerWeek` = (Σ miles of that shoe's runs within the last 28 days, boundary-inclusive at 28-days-ago, future runs excluded) ÷ 4. `milesRemaining = max(limit − currentMiles, 0)`. State machine: `currentMiles ≥ limit` → **"over"** (projectedDate null); else `recentMilesPerWeek ≤ 0` → **"inactive"** (projectedDate null — no divide-by-zero, never a bogus "never"); else `projectedDate = now + (milesRemaining / recentMilesPerWeek)·7 days` with state **"approaching"** when `currentMiles/limit ≥ 0.75` else **"ok"**. Returns `null` when no positive limit is set (no retirement target → nothing to project; the card omits the line). Pure / in-memory — `currentMiles` = `totalMileage` (startMileageOffset + assigned-run miles), `shoeRuns` derived from `shoeAssignedRuns`; no Firestore, no storage, no input mutation. The Shoes page renders one compact line under each card's mileage bar (`--caution`/`text-warning` when approaching, `--negative`/`text-danger` when over).

21. **Week Score + breakdown** (`src/utils/weekScore.ts`): `computeWeekScore(input)` is a weighted sum of three components, each `round(adherence × maxPoints)`, **pro-rated against the plan scheduled THROUGH TODAY** (not the full week) so an on-track mid-week reads as on-track instead of "incomplete". Inputs carry a `daysElapsed` (Monday-start → today inclusive, today = a full day, clamped 0..7; a fully-elapsed past week = 7, a future week = 0). Components: **Run miles** (`RUN_MAX_POINTS=40`, `min(actualToDate/plannedMilesThroughToday,1)`), **Training load** (`LOAD_MAX_POINTS=35`, `min(loadToDate/proRatedLoadTarget, LOAD_FULL_CREDIT_RATIO=1.2)/1.2` — full credit at 120% of the pro-rated baseline), **Workouts** (`WORKOUT_MAX_POINTS=25`, `min(completedToDate/sessionsScheduledThroughToday,1)`). **Pro-rating denominators**: planned-mile and session denominators sum only plan entries whose `weekday ≤ daysElapsed` (`isScheduledThroughToday`, computed at the page level from each plan week's entries); the load target is `proRatedLoadTarget(avgWeeklyLoad, daysElapsed) = avgWeeklyLoad × (daysElapsed/7)` — there is **no per-day planned load** in plan data, so this linear fraction of the 28-day weekly baseline is the load denominator. **Zero-denominator rule**: nothing scheduled through today for a component (rest day, future week, absent plan, or zero baseline) ⇒ that component scores **100%** so the card never looks "behind" for targets not yet due. Weights/caps/bands are unchanged from the prior full-week version; on the last day (`daysElapsed=7`) the pro-rated score equals the old full-week score by construction. `total = runScore + loadScore + workoutScore` (0–100). Label bands: ≥90 Excellent · ≥75 Strong · ≥60 Solid · ≥40 Light · else Recovery. Weights are exported constants so the scorer and the explainer can't drift. Helpers `daysElapsedInWeek`, `isScheduledThroughToday`, `proRatedLoadTarget`, `weekScoreBasisLabel` are pure exports. `buildWeekScoreBreakdown(input)` returns the per-component `{ key, label, unit, actual, target, earnedPoints, maxPoints, note }[]` + `total`, with `earnedPoints` taken **directly** from `computeWeekScore` (so `sum(earnedPoints) === total === computeWeekScore(input).total`; the load component's `target` is the pro-rated baseline so the explainer reconciles to the pro-rated score). The This Week card adds a small `basisLine` subtext stating the basis (e.g. "vs plan through Wed" / "vs full-week plan" / "Upcoming week") and renders the additive, collapsed-by-default "How is this calculated?" disclosure. The dashboard's full-week Running/Workout KPI tiles still show full-week planned values; only the Week Score is pro-rated.

22. **Plan date edit / resize / copy-with-new-start** (`src/utils/planDateEdit.ts`): pure date-recompute core for editing a plan's start date or length, and for copy-with-new-start-date (UI lands in later prompts). A plan stores only `startDate` (ISO, Monday-normalized by convention) + `weeks: PlanWeek[]`; there is **no stored end date or total-week count** — total weeks = `plan.weeks.length`, end date is derived. Entry calendar dates are always derived (`startDate + weekIndex*7 + (weekday-1)` days, mirroring `plannedEntryDate`), so a "slide" is literally changing `startDate`. Exports: `snapToMonday(iso)` (Monday on/before a date); `derivePlanEndDate(plan)` = `startDate + weeks.length*7 - 1` days (canonical version of the former inline `endDateForWeeks` in `plans/page.tsx`, to be repointed); `weeksForSpan(start, end)` = `max(1, ceil(daySpan/7))` (matches the create-modal ceil formula, DST-safe via UTC-component day diff); `slideStartDate(plan, newStart)` (snaps to Monday, weeks.length unchanged); `resizeToEndDate(plan, newEnd)` (start fixed, lengthen → append `{weekNumber, entries:[]}`, shorten → `slice`, equal → no-op, `newWeeks` clamped ≥1, no weekIndex reindex); `droppedWeeksWithEntries(plan, newWeeks)` (trailing weeks beyond `newWeeks` holding non-rest entries, for the shorten-confirm dialog); `copyPlanWithNewStart(plan, name, newStart)` (deep-copy then Monday-snapped slide → `createPlan` payload); `upcomingMonday(today)` (Monday on/after today — today when today is Monday — the copy modal's default start); and the race-alignment helpers `endsAfterRace(plan, raceDateIso)` + `raceAlignment(plan, raceDateIso, toleranceDays=RACE_ALIGN_TOLERANCE_DAYS=7)` → `"after" | "before" | "aligned"` (`"before"` only when the gap to the race exceeds the tolerance; no race date → `"aligned"`). **Policies:** start dates are Monday-snapped; **sliding a WORKOUT plan clears `completed`/`completedAt` on every entry** (workout completion is stored per-entry, so it must be reset so auto-match re-evaluates against the new dates) while running plans (live matching, no stored state) are left structurally identical. All functions are pure — no input mutation, local-date parsing only (no UTC off-by-one), no Firestore/IO.

23. **Route performance, trend windows & run impact** (`src/utils/routeClustering.ts`, `src/utils/routePerformance.ts`, `src/utils/routeTrendRuns.ts`, `src/utils/runImpact.ts` — all pure, no Firestore writes):
   - **Single clustering source**: `clusterRoutesGeographic` was extracted VERBATIM from the Routes page into `routeClustering.ts` (distance ±0.5 mi, then start points ≤ 300 m via `getRouteStartPoint`, distance-only fallback when a start point is missing; representative run = best pace). Both the Routes page and Run Detail consume this one algorithm — never fork it. `findClusterForRun(clusters, workoutId)` locates the current run's group.
   - **Best-effort ranking rule** (`computeRoutePerformance`, approved product decision): rank is by PACE across **ALL matched runs in the group, including distance-mismatched ones** (e.g. a 13.4 mi race clustered onto a 9 mi route still ranks by its pace) — clustering decides membership, ranking never re-filters by distance. Pace ties break by earlier date (the earlier run set the mark). Returns null when the run isn't in the group or the group has < 2 runs (card renders nothing). `routeAvgPaceSeconds` = arithmetic mean of matched paces; `deltaVsAvgSeconds` = current − avg (negative = faster). Row pace (`MatchedRunSummary.paceSeconds`) prefers the stored `avgPaceSecPerMile`, falling back to `durationSeconds/distanceMiles` (PR-computation convention); rows with no derivable pace are dropped. `load` resolves via `resolveDisplayLoad` (single source of truth; null renders "—").
   - **Trend window rule** (`selectRouteTrendRuns`, approved product decision): total ≤ 30 → all runs; else last-6-calendar-months when that window holds ≤ 20 (min 1 run — an idle route still shows its latest point); else last 10 runs. 24. **Weekly load bands** (`src/utils/weeklyLoad.ts`, pure — powers the Personal Insights Weekly Training Load tile): `buildWeeklyLoadModel(workouts, maxHr, restingHr, now)` buckets ALL activities (runs AND non-run workouts, matching the stacked chart it replaced) into Monday-start weeks (shared `weekStart`, local-date ISO keys per invariant #12) and returns (a) a 16-week display series (`WEEKLY_LOAD_WEEKS_BACK`, oldest → newest, zero-activity weeks present with total 0) and (b) `medianWeekly` — the median weekly total over the trailing **6 calendar months** (`WEEKLY_LOAD_MEDIAN_MONTHS`), counting only **completed** weeks (the in-progress week would bias the baseline down) with **≥ 1 activity** (zero-activity weeks excluded from the median by spec). ## Section 6 — Known Sharp Edges

1. **Per-point HR now exists on the `route` subcollection** (field `hr: number | null`, added iOS commit 84dfbf3). Run Detail HR-zone breakdown (`ZoneBreakdown`) and Training Load consumers now read `users/{uid}/settings/prefs.maxHeartRate` when set and otherwise fall back to `DEFAULT_MAX_HR=185` through `resolveMaxHr(settings)`. Per-mile HR is still sourced separately from the `mileSplits` subcollection (Section 3).

2. **Firestore subcollection security rules**: Every subcollection needs an explicit Firestore security rule. Missing rules produce permission-denied failures, which can be silent or surface as empty data. Always add rules when adding a new subcollection. (`src/lib/firebase.ts` comment)

3. **stravaActivities is not uid-scoped**: `fetchActivities()` in `src/services/activities.ts` queries the global `stravaActivities` collection with no uid filter. This is a legacy design. Do not add uid filtering without a migration plan.

4. **Legacy efficiencyRaw / efficiencyScore**: These fields are on the `HealthWorkout` type and in Firestore documents for backward compat with iOS, but no current UI reads them. Training Load (TRIMP) replaced the old efficiency metric. `src/types/healthWorkout.ts`

5. **STRENGTH_LOAD_FACTOR history**: The constant was 0.20 before being bumped to 0.25. Any analytics comparing load scores from before the bump against scores after will see a ~25% step-change for strength activities.

6. **Temporary debug console.log**: `getActivityContext()` in `src/utils/trainingLoad.ts` has an unflagged `console.log("[trainingLoad] activityType:", ...)` that was never removed after production validation.

7. **React hooks before early returns**: All hooks must appear before any conditional `return`. Guards like `if (!uid) return null` must come after all `useState`/`useEffect`/`useMemo` calls. Violating this causes React error #310.

8. **planType backward compat**: RunningPlan docs in Firestore have no `planType` field. Every `plans.ts` read path defaults to `"running"`. Any new code reading plans must apply this default.

8a. **Plan status vs isActive (dual-write, no write-back)**: `status` is the in-app source of truth for plan lifecycle; `isActive` is a kept-in-sync mirror of `status === "active"` that exists **only for the iOS reader** — do not remove it or stop writing it. Rules: (1) every writer that sets one field sets both (`createPlan`/copy/seed/`setActivePlan` + the Plans-page optimistic mirror); a record whose two fields disagree is a bug. (2) Read all in-app lifecycle decisions from `status === "active"`, never `isActive`. (3) Legacy docs missing `status` get it derived **in memory** at fetch time (`derivePlanStatus`); there is **no migration that writes status back on load** — old docs persist their old shape until a normal edit re-saves them. (4) `setActivePlan` (and its pure helper `nextStatusForSibling`) must **leave a same-type `"completed"` sibling unchanged** — activating another plan must never silently un-complete a finished one. (5) Adding `status`/`completedAt` needs no Firestore-rules change (scalar top-level fields on an existing owner-writable doc). (6) **Completion write semantics** (`planCompletionPatch` / `setPlanCompletion`): completing sets `{ status:"completed", isActive:false, completedAt: nowIso }`; reopening sets `{ status:"draft", isActive:false, completedAt: undefined }` — the `undefined` is dropped by `stripUndefined`, so reopen genuinely removes the field rather than writing null. Completion is **self-only** (no sibling writes), unlike `setActivePlan`. A completed plan still shows **Set Active** (status !== "active"), so it can be reactivated directly or Reopened to draft — both are valid paths.

9. **Legacy pilates plans**: Documents with `planType: "pilates"` exist in some Firestore instances. The code detects them and renders a "delete me" prompt. No write path creates them. `src/types/plan.ts:LegacyPilatesPlan`

10. **Firestore rules: repo file is canonical, deploy is MANUAL**: `firestore.rules` at the repo root is now the canonical source of truth (added 2026-06-10 from a console export + audit fixes). Editing the file does **nothing** until it is deployed — rules only go live via a manual deploy: `firebase deploy --only firestore:rules --project malignant-metro` (CLI configured via repo-root `firebase.json` + `.firebaserc` as of 2026-06-11; console paste remains a fallback — see CLAUDE.md "Deploying Firestore Rules"). Going forward: any new collection/subcollection needs its rule added to the **repo file first**, then deployed — never edit rules directly in the console without mirroring back to the repo, or the file goes stale. Until the 2026-06-10 file is deployed, `users/{uid}/healthGoals` and `users/{uid}/dismissedDuplicates` writes fail with permission-denied (both rules exist only in the repo file). 11. **Riegel NaN date passthrough (fixed)**: `buildQualifyingEfforts` once allowed runs with NaN `startDate` through the cutoff check (`NaN < cutoff` is false, so they passed). Fixed with `!isFinite(startMs)` guard. `src/utils/riegelFit.ts`

12. **GAP basis & moving-time**: As of the ratio-based GAP rewrite (Section 5 item 12), **GAP is anchored to the displayed `avgPaceSecPerMile`** (GAP = pace × grade ratio), so flat ground gives GAP == Pace exactly — the old "pace uses elapsed, GAP uses moving, so they differ by design" divergence no longer applies to the flat case. `computeRunGap` still derives the moving-time/distance basis from route points (segments ≥ `MIN_MOVING_SPEED_MS=0.5` m/s) for the grade ratio and the per-point/per-mile series; if route timestamps are degenerate (< `MIN_DERIVED_MOVING_SEC=60` s of derivable moving time) it falls back to the passed `durationSeconds` with a `console.warn` and treats every real-distance segment as moving. `src/utils/gradeAdjustedPace.ts`

Both the run listing page and the run detail page resolve a run's shoe via `{ ...autoAssigned, ...manualMap }` (manual wins). The detail page does so through the `useResolvedShoeAssignment` hook (`src/hooks/useResolvedShoeAssignment.ts`). Dirty-state is corrected for both: `useUnsavedChanges` is passed a **real `hasUnsavedChanges` flag** (true only while a mutation's autosave write is in flight, cleared in a `finally`), **not** the edit-mode boolean. Entering edit mode, or opening a row editor and clicking Cancel, no longer arms the browser "leave site?" warning. One consequence: a brief dirty window now also exists for view-mode writes (workout mark-complete / notes), where the guard was previously inert in view mode. **Initial week / deep-link:** `PlanEditor` takes an optional `initialWeekIndex` (clamped via `resolveInitialWeekIndex`) — absent → active-plan current week, else week 0; the Plans page keys `RunningPlanDetail` by `planId-weekIndex` so a deep-link re-lands on its week while in-editor pagination never remounts (the target week is set when a plan is picked from the list). **Update:** the calendar running-pill no longer deep-links into the editor — `onRunningEventClick` was removed from `CalendarView`; running pills now open `RunActivityModal` instead (see §6 #20). **Per-week progress bar** is restored via an optional `config.renderWeekAccessory` slot (running supplies the planned-vs-matched completion bar from `computeWeekCompletion`; workout omits it, so `PlanEditor` stays type-agnostic). **Single edit entry point:** the running header has one "Edit" toggle (no separate rename pencil — the plan name is inline-editable in edit mode, persisted on blur); there is **no "Save All"** anywhere (persistence is autosave-per-mutation). Both earlier regressions (progress bar, calendar deep-link) are **RESOLVED**.

16. **Responsive shell breakpoint convention (`lg` = desktop/mobile divider)**: The app shell swaps navigation chrome at Tailwind's `lg` breakpoint — `< lg` shows the fixed bottom tab bar + "More" sheet and **hides** the sidebar (`hidden lg:flex`); `>= lg` shows the sidebar and **hides** the tab bar (`lg:hidden`). Future responsive work (content reflow, Builds 2 & 3) must respect this same `lg` divider so chrome and content agree on what "mobile" means. `<main>` reserves bottom space for the fixed bar via an inline `paddingBottom: calc(5rem + env(safe-area-inset-bottom))` (so iOS safe-area is honored); do not remove this or content hides behind the bar on mobile. Nav items are a single source of truth in `src/components/layout/navItems.ts` — never hard-code a nav list in a new component; import from there so the sidebar and tab bar/More cannot drift (guarded by `navItems.test.ts`).

17. **Mobile content-reflow patterns (`< lg`)**: The data pages reflow below `lg` using three established patterns — match them for any new wide layout rather than inventing a fourth: (a) **wide week/calendar grids** → wrap a `min-w-[…] grid grid-cols-7` in `overflow-x-auto` (horizontal scroll strip), as `WeekCalendar` does — NOT a forced 7-col squeeze; (b) **KPI/stat tile rows** → responsive column counts (`grid-cols-2 sm:grid-cols-3 lg:grid-cols-N`), never a fixed `grid-cols-6` base; (c) **side-by-side rail + detail panes** → a `mobileView: "list" | "detail"` state toggling `hidden lg:flex` on each pane (rail full-width `w-full lg:w-64`, `lg:hidden` back button), as the Plans page does. The `mobileView` hook must be declared with the other Plans hooks BEFORE any early return (React #310). Every such pattern is `lg:`-gated so `>= lg` is byte-identical to the desktop-first layout; the reflow is additive at `< lg` only. Compact fixed grids that already fit a phone (e.g. 3-up stat triplets, mini bar charts) are intentionally left un-prefixed. (d) **wide data tables** → wrap the real `<table className="w-full text-sm">` in `overflow-x-auto` (horizontal scroll strip), as Plan Insights "Performance by Run Type", Personal Insights PR table, the Health metrics table, and `MileSplitsTable` all do. `w-full` keeps the table inside the viewport (so short-valued tables just compress and rarely need to scroll), while `overflow-x-auto` is the safety net that scrolls instead of clipping when content is wide — no columns are hidden/dropped on mobile (info is never truncated) and no JS/view-state is needed (pure CSS). Prefer the scroll strip over card-stacking for these because their value is cross-column comparison. All four tables and every pattern above are `lg:`-gated / additive at `< lg`, so `>= lg` is byte-identical to the desktop-first layout.

18. **Route availability is derived from DATA, not the `hasRoute` flag alone (run detail page)**: The iOS headless background sync can write a populated `route` subcollection but be suspended before the trailing `hasRoute: true` flag write lands — leaving a real GPS run flagged `hasRoute: false` (and `hasRouteHR` absent). `runs/[id]/page.tsx` previously gated the route + `mileSplits` fetch on `if (w?.hasRoute)`, so such a run rendered with **no map, no splits, no HR** even though the data existed. The page now **always attempts the `route` read** and derives availability from the result via `src/utils/routeAvailability.ts`: `isRoutePresent(count)` (≥ `MIN_ROUTE_POINTS = 2`) gates the dependent fetches (best-efforts compute + `mileSplits`/per-mile-HR), and `deriveEffectiveHasRoute(flag, count) = flag || count ≥ 2` drives ALL downstream rendering (map, `MileSplitsTable`, `MileSplitCharts`, `RunOverlayChart`, `ZoneBreakdown`). A genuinely route-less workout (0–1 points) is unchanged — it reads 0 points and shows "No GPS route available for this run". Cost: one extra `route` read for non-route workouts opened in the detail view. **Data repair**: `scripts/repairHasRoute.ts` (env-gated Vitest runner `scripts/repairHasRoute.test.ts`, same harness as the training-load backfill — `REPAIR=1` dry-run / `REPAIR=commit` apply) finds run-like docs where `hasRoute ≠ true` but the `route` subcollection is non-empty and sets `hasRoute: true` (+`hasRouteHR: true` only when probed route points carry an `hr` field), via owner `setDoc(merge:true)`. The iOS root cause self-heals on the next wake (MEA commit 19f647a); the web fallback + repair script are the belt-and-suspenders web-side guards. **`routeComplete` awareness (display-hint only)**: iOS also writes a `routeComplete` marker on the parent doc (`true` = full route written, `false` = partial/still completing on a later wake, **absent on legacy docs → treated as complete**). The web reads it (`HealthWorkout.routeComplete`, mapped tri-state in `docToHealthWorkout`) and, via `isRouteSyncing(routeComplete) === (routeComplete === false)`, shows a subtle "Route still syncing — showing the portion received so far." caption under the map. This is **purely cosmetic** — a partial route still renders every point it has, and route availability is NEVER gated on `routeComplete` (a `false`/absent marker never hides a route). The web does not write `routeComplete`.

19. **Run weather is fetched client-side from the `route` subcollection coord, not the doc**: The run detail page (`runs/[id]/page.tsx`) backfills `healthWorkouts/{id}.weather` (a `WeatherSnapshot`) on load — but only when the `route` subcollection has ≥ 1 point, because **there is no `startLat`/`startLng` on the parent doc**; the start coordinate is `routePoints[0].lat/lng` (same source `routeCache.ts` uses). `fetchWeatherForRun` (`src/lib/weather.ts`) calls the Open-Meteo archive API with **no `timezone` param** (so hourly times are UTC) and selects the slot that **contains** the run's start instant — floor-to-the-hour (6:42 → 6:00, NOT round-to-nearest) — returning `null` on any failure (network / non-2xx / malformed / missing slot) so the tile silently hides. Persisted via `saveWeatherForWorkout` owner `setDoc(merge:true)` — no new Firestore rule. Already-stored weather is reused (no refetch). Cross-timezone runs (run location ≠ browser TZ) can match an adjacent UTC slot; acceptable for a display tile.

20. **Planned-vs-actual run modal (`RunActivityModal`, shared by two surfaces)**: A shared modal (`src/components/runs/RunActivityModal.tsx`) opens from a planned **running** session on the This Week dashboard's Running Plan tile (`PlanProgressCard` rows — rest days excluded) and the Plans page **Calendar** tab (running pills in `CalendarView`, week + month). It always shows planned distance/pace/notes, plus — when the session is matched to a real run — an Actual block (distance/pace/avg HR/duration), a **Mile | Pace | HR** splits table, and a "View Full Run →" link to `/runs/[id]`. Completion badge is green "Completed" when matched, muted "Upcoming" when not. **Per-mile pace is GPS-derived, NOT from the `mileSplits` subcollection** (which carries only `avgBpm`/`sampleCount` — §3): on open, and only when matched, the modal lazily fetches the run's `route` subcollection + `mileSplits` subcollection and reuses `computeMileSplits(routePoints, avgHeartRate)` plus the per-mile `avgBpm` merge — the exact path `runs/[id]/page.tsx` uses; the table is omitted when the run has no GPS, and the HR column self-hides when no per-mile HR exists. **Read-only — no Firestore writes.** The matched run is resolved from the **same `matchPlanToActual`** that drives the row/pill status (dashboard reuses the tile's existing `matchMap`; the calendar memoizes a per-plan `matchMap` in `CalendarView`, keyed off the new `CalendarEvent.entryId`), so the modal's matched/upcoming state can't disagree with the row icon / pill ✓. **Behavior change:** the Plans calendar running-pill previously deep-linked into the plan editor week (`onRunningEventClick`, now **removed** from `CalendarView` — see §6 #15); it opens this modal instead. Workout pills are unchanged (still navigate to the workout follow-along page).

21. **HR-anchor change recomputes ALL stored loads — use `recomputeAllTrainingLoad`, NOT the runs-only `backfillTrainingLoad`**: Saving a changed max/resting HR on `/settings` triggers a client-side recompute over every HR-bearing workout of ANY type and rewrites `trainingLoadV2`/`trainingLoadMethod` (`merge:true`) under the NEWLY-saved anchors (Section 5 item 4a). Two gotchas: (a) **do not** reuse `backfillTrainingLoad(uid, settings)` for anchor-change recompute — it filters `isRunLike == true` and would silently skip HIIT/OTF/strength/Pilates (the sessions most sensitive to the anchors); `recomputeAllTrainingLoad` has no type filter and is the correct entry point. (b) It can issue many reads/writes on a single save (one list query + one `getDoc`/`setDoc` per HR-bearing workout, since it reuses `computeAndStoreTrainingLoad` verbatim rather than re-fetching once) — it is idempotent and safe to re-run, and a failed recompute leaves already-written docs intact (settings remain saved). Stored values written under old anchors *before* this feature existed stay stale until the user next changes an anchor (or a backfill runs).

22. **Enrich-on-load: store-early-then-upgrade + the snapshot-loop guard** (`useEnrichTrainingLoads`, `enrichTrainingLoads`, `shouldEnrichLoad`/`enrichBasisKey` — Section 5 item 4a): iOS writes each `healthWorkouts` doc in two phases (base doc first, then `avgHeartRate` + the `hrStream` subcollection in a LATER write) and never writes a load, so a freshly-synced workout legitimately exists for one or more sync cycles with no stored load and sometimes no HR basis at all. Enrich-on-load auto-stores the load once the workout loads in the app. Three things to keep in mind: (a) **store-early-then-upgrade** — a workout whose `hrStream` hasn't landed yet is stored from its avg HR (`method:"avg-hr-fallback"`); because `resolveDisplayLoad` prefers a finite stored value FOREVER, that early value would permanently mask the better streamed value UNLESS it is re-enriched. The UPGRADE branch of `shouldEnrichLoad` does exactly that — when `hasRoute`/`hasHRStream` later flips true it re-runs once and rewrites `method:"streamed"`. (b) **snapshot loop** — the enrich write lands in the page's own `onSnapshot` query, which re-fires and re-runs the effect on its own output. The guard is `enrichBasisKey(workout)` = `id|hasRoute|hasHRStream`: each workout's basis is recorded the moment it's queued, so the same `(workout, basis)` is never enriched twice. This is what stops a compute that legitimately yields **null** (no usable HR — e.g. `hasHRStream` true but a sparse/empty stream and no avgHR) from re-writing the same null forever; a workout only re-qualifies when its basis genuinely changes (the stream arrives), which is the one upgrade we want. An `inFlightRef` additionally blocks overlapping passes. (c) **settling guard** — the hook waits until `settings !== undefined` (the profile fetch has resolved) before enriching, so it never stores loads against the DEFAULT HR anchors and then masks the correct value; a resolved `null` profile is fine (defaults are then intentional, and a later anchor change recomputes via §6 #21). **Coverage caveat**: each page only enriches the set IT loads — Workouts (non-runs) and Runs (runs), and only the loaded window (limit 500). A workout in neither loaded set isn't auto-stored until one of those pages loads it; its display is unaffected (live `resolveDisplayLoad`), and the settings-recompute / backfill remain the bulk write paths. Real Watch flush timing is only observable on-device.

23. **Two-pass iOS sync (resume) → stale `trainingLoadV2` (FIXED)**: A streamed load computed during a partial route/`hrStream` (iOS wrote the first ~N points, then RESUMED to completion on a later wake) is now recomputed when the extent/`routeComplete` signal changes; previously the enrich guard keyed only on `hasRoute`/`hasHRStream` booleans (unchanged across both passes) and `routeComplete` was ignored by the compute/enrich path, leaving a collapsed score (e.g. 6/16: stored **14** vs the correct **187**) stuck forever because a stored `method:"streamed"` value was otherwise terminal (§6 #22). The fix (`shouldEnrichLoad`/`enrichBasisKey` in `src/utils/trainingLoad.ts`, `computeAndStoreTrainingLoad` + `docToHealthWorkout` in `src/services/healthWorkouts.ts`, `HealthWorkout.trainingLoadBasisComplete` type): (a) `computeAndStoreTrainingLoad` records `trainingLoadBasisComplete = (routeComplete !== false)` — the route-completeness at compute time — alongside `trainingLoadV2`/`trainingLoadMethod`; (b) `shouldEnrichLoad` adds a RECOMPUTE branch (c): a stored `"streamed"` load re-qualifies when `routeComplete === true && trainingLoadBasisComplete === false` (the route has completed since the partial basis it was computed from); (c) `enrichBasisKey` now appends `routeComplete` → `id|hasRoute|hasHRStream|routeComplete`, so the partial→complete transition changes the loop-guard key and the one recompute pass is allowed through. **Non-thrashing**: the recompute writes `basisComplete=true`, so the branch is false on the next load — it fires at most ONCE per completion; healthy complete loads (`basisComplete` true) and legacy docs (`basisComplete` ABSENT → `undefined`, never `=== false`) are never re-run, so there is no deploy-time write burst. **Limitations**: there is no route/stream point-count field on the doc, so `routeComplete` is the only completion signal available (a count would be folded into `enrichBasisKey`/the basis if iOS ever adds one); the signal is route-specific, so a non-route `hrStream`-only workout (OTF/HIIT) that syncs in two passes is NOT auto-recomputed by this path; and a doc still mid-two-pass-sync at deploy time (partial load written by OLD code, basis ABSENT) won't auto-recompute on completion. All residual gaps are covered by the manual backfill (`scripts/backfillTrainingLoad.ts`, incl. `BACKFILL_STALE_ONLY=1`); existing collapsed runs were already repaired by that backfill, and this fix prevents RECURRENCE for future two-pass route runs.
